**internal/profile/profile.go:**

Package declaration: `package profile`

Imports: `bytes`, `fmt`, `strings`, `github.com/BurntSushi/toml`

Define types:

```go
// Frontmatter represents the TOML frontmatter of a profile file.
type Frontmatter struct {
    Model       string `toml:"model"`
    Description string `toml:"description"`
}

// Profile represents a parsed profile with metadata and system prompt.
type Profile struct {
    Name         string      // Derived from filename (set by Load, not Parse)
    Frontmatter  Frontmatter // Parsed TOML frontmatter
    Provider     string      // Extracted provider from "provider/model" (empty if bare model)
    ModelName    string      // Extracted model name from "provider/model" or bare model
    SystemPrompt string      // Markdown body after frontmatter
}
```

Implement `Parse(data []byte) (*Profile, error)`:
1. Convert data to string
2. Define `delimiter := "+++"`
3. Find first `+++` — if not found or doesn't start at line beginning, return `fmt.Errorf("missing frontmatter: no +++ delimiters found")`
4. Find second `+++` after first — if not found, return `fmt.Errorf("missing frontmatter: unclosed +++ delimiter")`
5. Extract TOML content between first and second `+++` lines
6. Decode TOML into `Frontmatter` using `toml.Decode(tomlContent, &fm)` — if error, return `fmt.Errorf("parsing frontmatter: %w", err)`
7. Extract body after second `+++` line, trim leading/trailing whitespace with `strings.TrimSpace()`
8. Split `fm.Model` on `/` using `strings.SplitN(fm.Model, "/", 2)`:
   - If contains `/`: provider = parts[0], modelName = parts[1]
   - If no `/`: provider = "", modelName = fm.Model
9. Return `&Profile{Frontmatter: fm, Provider: provider, ModelName: modelName, SystemPrompt: body}`

The parsing logic should work line-by-line: split content on `\n`, find lines that are exactly `+++`, extract TOML between them, body is everything after the second `+++` line.

| Boundary | Description |
|----------|-------------|
| filesystem → profile parser | Profile .md files read from user-owned config directory |
| profile name → file path | User-supplied profile name mapped to filesystem path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering | profile.Parse | accept | Files are in user-owned ~/.config/fenec/profiles/ — user controls their own profiles. No integrity concern. |
| T-16-02 | Information Disclosure | profile.Load | accept | Profile files may contain system prompts — but they are the user's own files in their own config directory. No cross-user risk. |
| T-16-03 | Elevation of Privilege | profile.Load | mitigate | Path traversal via profile name (e.g., "../../etc/passwd") — mitigated by validating name contains no `/`, `\`, or `.` characters in Load() before constructing filepath. |
| T-16-04 | Denial of Service | profile.Parse | accept | Malformed TOML or very large files could slow parsing — BurntSushi/toml handles this safely with error returns. Profile files are user-created and user-sized. |
</threat_model>
